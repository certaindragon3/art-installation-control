# Phase 9 Closeout: Unique Receivers + Animated Map Movement

Date: 2026-04-14

Production URL tested: `https://artinstallation.certaindragon3.work`

Note: the originally supplied `http://artinstallation.certaindragon3.work/` URL was blocked by the browser automation environment with `net::ERR_BLOCKED_BY_CLIENT`, so production browser testing used the HTTPS origin for the same Zeabur deployment.

## Delivered

- Receiver registration now resolves every live receiver to a unique final ID. If two browser instances request the same ID, the later instance receives a numeric suffix such as `phase9dup2`.
- Receiver tabs send a `clientInstanceId` so ordinary refresh/reconnect can reclaim the same ID instead of creating a new suffix.
- Map state now supports a `movement` payload with start position, target position, duration, loop flag, and server-assigned `startedAt`.
- Receiver pages interpolate the marker locally in the browser. Unity or HTTP callers only need to send one start/target command instead of streaming per-frame positions.
- Instant map position remains supported and clears any active movement, so manual placement/debug controls still work.
- Controller UI now exposes start X/Y, target X/Y, duration, loop, send movement, and stop movement controls.
- Professor-facing command docs include copyable movement examples.

## API Changes

### Socket.IO: Receiver Registration

`register_receiver` now accepts an optional browser-instance identity:

```typescript
interface ReceiverRegistration {
  receiverId: string;
  label?: string;
  clientInstanceId?: string;
}
```

Behavior:

- `receiverId` remains the requested ID from `/receiver/:id`.
- `clientInstanceId` is generated by the receiver browser tab and persisted in `sessionStorage`.
- The server may assign a different final `receiverId` when the requested ID is already occupied.
- The final ID is returned through the normal `receiver_state_update` payload and appears in controller receiver lists.

Example:

```json
{
  "receiverId": "screen-a",
  "label": "Receiver screen-a",
  "clientInstanceId": "4f62c7de-4ab5-4e23-a0da-b747bf6a85aa"
}
```

If another active browser instance requests `screen-a`, it can be assigned `screen-a2`. Commands must target the final assigned ID, not the originally requested ID.

### Receiver State: Map Shape

`ReceiverState.config.map` now includes `movement`:

```typescript
interface MapMovementConfig {
  fromX: number;
  fromY: number;
  toX: number;
  toY: number;
  startedAt: string;
  durationMs: number;
  loop: boolean;
}

interface MapConfig {
  visible: boolean;
  enabled: boolean;
  playerPosX: number;
  playerPosY: number;
  movement: MapMovementConfig | null;
}
```

This affects:

- Socket.IO `receiver_state_update`
- Socket.IO `receiver_list`
- `GET /api/controller/receivers`
- `POST /api/controller/command` responses, because they include updated receiver snapshots

### HTTP / Socket.IO Command: Start Map Movement

Use the existing unified command path with `command: "set_module_state"` and `payload.module: "map"`.

Endpoint:

```http
POST /api/controller/command
Content-Type: application/json
```

Payload:

```json
{
  "command": "set_module_state",
  "targetId": "screen-a2",
  "payload": {
    "module": "map",
    "patch": {
      "visible": true,
      "enabled": true,
      "movement": {
        "fromX": 0.1,
        "fromY": 0.8,
        "toX": 0.9,
        "toY": 0.2,
        "durationMs": 20000,
        "loop": true
      }
    }
  }
}
```

Socket.IO controllers send the same object through `control_message`, with the normal `timestamp` field added by the controller client.

Server normalization:

- Coordinates are normalized `0..1`.
- `fromX` / `fromY` can also be sent as `startX` / `startY`.
- `toX` / `toY` can also be sent as `targetX` / `targetY`.
- Missing coordinate fields fall back to the receiver's current map position.
- Coordinates outside `0..1` are clamped.
- `durationMs` defaults to `20000`.
- `durationMs` is clamped to the server's allowed movement duration range.
- `loop` defaults to `true`.
- `startedAt` is optional; when omitted, the server writes the current ISO timestamp.
- When movement is accepted, `playerPosX` and `playerPosY` are updated to the target position so controller snapshots show the final target.

Response excerpt:

```json
{
  "ok": true,
  "deliveredReceiverIds": ["screen-a2"],
  "receivers": [
    {
      "receiverId": "screen-a2",
      "config": {
        "map": {
          "visible": true,
          "enabled": true,
          "playerPosX": 0.9,
          "playerPosY": 0.2,
          "movement": {
            "fromX": 0.1,
            "fromY": 0.8,
            "toX": 0.9,
            "toY": 0.2,
            "startedAt": "2026-04-14T14:33:43.461Z",
            "durationMs": 20000,
            "loop": true
          }
        }
      }
    }
  ]
}
```

### HTTP / Socket.IO Command: Stop Map Movement

To stop interpolation without hiding the map:

```json
{
  "command": "set_module_state",
  "targetId": "screen-a2",
  "payload": {
    "module": "map",
    "patch": {
      "movement": null
    }
  }
}
```

### HTTP / Socket.IO Command: Instant Map Position

Instant position remains supported and now explicitly clears active movement:

```json
{
  "command": "set_module_state",
  "targetId": "screen-a2",
  "payload": {
    "module": "map",
    "patch": {
      "playerPosX": 0.2,
      "playerPosY": 0.8
    }
  }
}
```

Compatibility aliases:

- `x` can be used instead of `playerPosX`.
- `y` can be used instead of `playerPosY`.

### No Per-Frame API

There is intentionally no new per-frame position API. Unity should send one movement command per segment; receiver browsers interpolate locally from `movement.startedAt`, `durationMs`, and `loop`.

## Production Notes

- Keep Zeabur replica count at 1. Receiver/controller state is still in memory and is intentionally not horizontally scalable.
- The production environment had other live/offline receivers from earlier test sessions. Phase 9 smoke used the unique `phase9dup` test IDs to avoid mixing results.
- For CLI `curl` against Zeabur HTTPS, `--http1.1` was more reliable than the default HTTP/2 path in this environment. Browser fetches and Socket.IO testing were unaffected.
- Receiver summary cards are rendered as buttons, but the reliable selection path in the smoke test was the controller combobox.
